## Definite bugs

### 1. Unbounded goroutine + memory leak via never-closed event channels

**Status:** ✅ Fixed in `653b18a` (`fix: close never-closed event channels to stop goroutine leak`).

**Severity: definite bug (medium impact) · Confidence: high**

- **Where:** `internal/lobby/model.go:345` (`waitForPhoneEvent`) and `internal/phone/app.go:894` (`waitForChar`); root cause in `internal/registry/registry.go:96` (`Unregister`) and `internal/phone/call.go:239` (`Hangup`).
- **What:** Both `waitForPhoneEvent` and `waitForChar` run a **blocking** channel receive (`event, ok := <-ch`) inside a `tea.Cmd` goroutine. The channels they block on — the registry's per-account `entry.notify` (created in `Register`) and PHONE's per-participant `Participant.IncomingChar` (created in `Dial`/`Answer`) — are **never closed anywhere**. Confirmed by grep: every `close()` in the codebase is on a ring/stop channel (`stopRing`, `stop`, `existing`), never on `notify` or `IncomingChar`. `Unregister` just `delete`s the map entry; `Hangup` just drops the participant from the slice.
- **Failure scenario:** A user connects → reaches the lobby → one `waitForPhoneEvent` goroutine is armed on their `notify` channel. They disconnect → `Unregister` removes the entry but doesn't close the channel → the goroutine stays blocked on `<-ch` **forever** (a blocked goroutine is a GC root, so it and its buffered channel are never reclaimed). Same for every PHONE call: each participation leaves one `waitForChar` goroutine permanently blocked on the departed call's `IncomingChar`. Accumulates one leaked goroutine per lobby session **plus** one per PHONE call participation, for the entire uptime of the server process. On a 24/7 multi-user BBS with connection/call churn this grows without bound.
- **Smoking gun:** both receivers already handle `event, ok := <-ch; if !ok { return nil }` — they were written expecting a channel *close* that the code never performs. The cleanup was clearly intended and dropped.
- **Fix caveat (for later):** the naive fix (close the channel in `Unregister`/`Hangup`) races with the non-blocking sends in `sendEvent`/`BroadcastChar` and would panic on send-to-closed-channel. A correct fix needs a coordinated shutdown signal — e.g. a per-session `done` channel selected alongside the receive, or closing under the same lock the senders hold. Not a one-liner.

> ★ Insight
> This is the canonical Bubble Tea footgun: a `tea.Cmd` that does a blocking read
> leaks its goroutine if the channel neither produces nor closes, because Bubble
> Tea has no way to cancel an in-flight Cmd. The `if !ok` handling is the
> fingerprint of an intended-but-missing close. It's the one finding here worth
> actually prioritizing — it's a real availability/stability issue for a service
> meant to stay up.

### 2. Invite-code use is consumed even when registration is then abandoned or fails

**Status:** ✅ Fixed in `df150e8` (`fix: consume invite code only on confirmed account creation`).

**Severity: definite bug (low impact) · Confidence: high**

- **Where:** `internal/registration/app.go:163` (`ValidateAndConsumeInvite` at the `stateInviteCode` step) vs. `:208` (`CreatePendingAccount` in `finalise`).
- **What:** The invite use is decremented at the invite-code step, several states *before* the account is actually created. If the user then Ctrl+C's during password entry (`:74`), or `CreatePendingAccount` fails (e.g. username taken in the race window, `:209`), the invite use is already burned with no account created.
- **Failure scenario:** A 1-use invite code is entered correctly; the user disconnects at the password prompt. The code is now spent — the legitimate invitee can never use it. For invite-only communities handing out single-use codes, this silently voids codes.

## Plausible risks worth checking

### 3. `lastUsableAdminGuard` is a non-atomic check-then-act (TOCTOU)
**Severity: plausible risk (low likelihood) · Confidence: high on the race, low on real-world reachability**

- **Where:** `internal/lobby/commands.go:905` (`lastUsableAdminGuard`), used by `banCommand:1057` and `deleteUserCommand:1328`.
- **What:** The guard reads `CountUsableAdmins()` and then the caller proceeds to `BanUser`/`DeleteUser` as separate, unsynchronized operations.
- **Failure scenario:** Two admins (two sessions/goroutines) each BAN the other near-simultaneously. Both call the guard, both read count = 2 (`usable > 1`, passes), then both mutate → zero usable admins, exactly the state the guard exists to prevent. Requires two concurrent admins acting in the same instant, so it's rare — but it defeats the guarantee. A correct fix makes the check-and-mutate atomic in one conditional SQL `UPDATE`.
- **Note:** The single-admin and sequential cases are handled correctly — only BAN and DELETE USER can reduce the usable-admin count, and both carry the guard. This is purely the concurrency gap.

### 4. Read-side time correctness is latent on the SQLite DSN never gaining a `_timezone` param
**Severity: plausible risk (latent) · Confidence: high**

- **Where:** `internal/store/store.go:78` (`sql.Open("sqlite", path)`), interacting with `IsUsableAdmin:64`, the lockout checks at `cmd/server/main.go:473` / `internal/setpassword/setpassword.go:142`, and `commands.go:1302`.
- **What:** Traced the driver (`modernc.org/sqlite@v1.53.0`, `conn.go:162`): a scanned naive `DATETIME` string is parsed with `time.Parse` (→ UTC) **only because `c.loc == nil`**, which holds only because the DSN has no `_timezone`/`_loc` query param. Every timestamp is written as naive-UTC and read back as UTC, so all read-side comparisons are currently **correct** (this is the good news — see clean results below). But the correctness is invisible and fragile: if anyone ever appends `?_timezone=Local` (or similar) to that DSN, `c.loc` becomes non-nil, the driver reinterprets stored UTC strings as local, and every lockout/ban read-side comparison silently skews by the server's UTC offset — reintroducing exactly the class of bug just fixed on the write side.
- **Suggested guard (later):** a one-line comment at the DSN and a regression test that stores a known instant and reads it back, so a future DSN change trips a test rather than shipping silently.

### 5. Reserved username `new` is creatable by admins (bypasses the self-registration block)
**Severity: plausible risk (low) · Confidence: high**

- **Where:** `internal/lobby/commands.go:1392` (`validateNewUsername`) and `cmd/adduser/main.go:34` — neither blocks `new`; only `internal/registration/app.go:312` (`reservedUsernames`) does.
- **What:** `CREATE USER new` or `adduser -username new` succeeds. A `user`-role "new" account is unreachable (the public listener routes username `new` to registration, `main.go:401`), so it's a zombie. An `admin`-role "new" *is* reachable on the admin listener (which doesn't special-case `new`). Deliberate for admins per the comment ("skips the reserved-word block"), but `new` specifically collides with the registration sentinel and is worth an explicit exclusion. Admin/operator footgun, not remotely exploitable.

### 6. Invite expiry fail-opens if the stored timestamp can't be parsed
**Severity: plausible risk (low) · Confidence: medium**

- **Where:** `internal/store/store.go:629-636` (`ValidateAndConsumeInvite`).
- **What:** If both `time.Parse` layouts fail, `err != nil`, so the `if err == nil && ... time.Now().After(t)` expiry check is skipped and the invite is treated as **valid**. Only reachable with a corrupted/hand-edited `expires_at` (all normal writes use a known format), so low likelihood — but a parse failure fails *open* (never-expires) rather than closed.

## Minor / stylistic

- **Display-zone inconsistency (`internal/lobby/commands.go`):** `TIME`/`WHO` render **local** server time (`:605`,`:621`), but `FINGER` last-login (`:704`), `LIST PENDING` (`:986`), and `LIST USERS` (`:1292`) render **UTC** (values scanned as UTC). On a non-UTC server these visibly disagree (e.g. "last login 19:00" vs. a "TIME" of 15:00). Cosmetic, not a correctness bug. Confidence: high.
- **Inconsistent nil-guards:** `listUsersCommand:1278` and `kickCommand:1025` call `m.db`/`m.reg` without the `== nil` guard their sibling handlers use; a nil would panic (caught by `dispatch`'s `recover`, so contained). Stylistic inconsistency. Confidence: high.
- **`requireAdminLogged` before arg validation:** `banCommand:1041` logs the admin action before the `len(parts) < 2` check at `:1048`, so a malformed `BAN alice` (missing duration) emits an audit line then returns usage. Matches the documented "log the attempt" intent; just adds audit noise. Confidence: high.
- **`preAuthTimeout` timer not stopped (`cmd/server/main.go:372`):** `time.After(d)` isn't `Stop()`'d on the auth-success/ctx-done paths, so the timer lives until `d` elapses even after auth completes. Bounded (≤ `AUTH_TIMEOUT_SECONDS`) and self-clearing — a minor holding, not a leak. Confidence: high.
- **`generateInviteCode:1266` modulo bias:** `int(b[0]) % len(...)` is slightly biased, negligible for a ~144k-code space and not security-relevant. Confidence: high.
- **`cmd/adduser -password` plaintext on CLI:** visible in `ps`/shell history. It's the documented bootstrap tool and an accepted tradeoff; noting for completeness. Confidence: high.
- **`Invite.IsExpired()` (`store.go:171`) is dead code** (already flagged in open-questions.md). Confidence: high.

## Verified clean (the areas most worth confirming)

- **Time handling — no remaining UTC/local bug.** Every timestamp is written naive-UTC (`datetime('now')`, `CURRENT_TIMESTAMP`, or explicit `.UTC().Format()`) and read back as UTC (driver `c.loc == nil` → `time.Parse`; explicit `time.Parse` in the invite path). Go's `.Before`/`.After` compare absolute instants regardless of a `time.Time`'s display location, so `IsUsableAdmin`, both lockout checks, and the SQL string comparisons (`banned_until < datetime('now')`, both sides UTC and fixed-width sortable) are all correct. The `BanUser`/`CreateInvite` fix closed the last gap. (Caveat #4 above is the only residual.)
- **SQL construction — clean, high confidence.** All 34 DB call sites live in `internal/store`, all use `?` bound parameters. The only `fmt.Sprintf`s near SQL build `datetime` *modifier values* (`"+300 seconds"`) passed as bound params — never query text. No user-controlled value is ever concatenated into a query.
- **Dual-listener auth + timing.** Public refuses admins and admin refuses non-admins *before* password verification; every rejection path runs exactly one argon2id call (real or the matching-params `dummyHash`), so there's no username-existence or role timing side channel. Lockout is correct: rejected before argon2id while actively locked, and `RecordFailedAttempt` is *not* called on the locked path (no permanent-lock escalation).
- **Role-gating / command hiding.** `adminCommandKeys` (derived from `adminHelpTopics` + the manual `INVITE`) covers every admin entry in both `commands` and `argCommands` — all enumerated. Non-admins get identical "not a recognized command" for admin verbs and typos alike, including `HELP <admin-topic>`. `SET PASSWORD` correctly stays out of the admin set. Every admin handler also has its own `requireAdmin`/`requireAdminLogged` behind the dispatch gate.
- **Guard parity.** Only `BAN` and `DELETE USER` can reduce the usable-admin count; both carry `lastUsableAdminGuard` (modulo the TOCTOU in #3). `DELETE USER` also blocks self-deletion unconditionally.
- **PHONE concurrency (call.go).** Careful mutex serialization, non-blocking sends everywhere (`sendEvent`/`BroadcastChar` use `select … default`), and correct `stopRing` double-close avoidance. This is why the leak in #1 is a *never-closed* problem, not a *double-close* one.
- **Secrets/logging.** The startup config dump (`main.go:112`) is a hand-built format string with no secret field; `BOOTSTRAP_ADMIN_PASSWORD` is read via `os.Getenv` and never enters the `config` struct. No password, hash, or invite code is logged on any auth/bootstrap path. Masked-input flows keep plaintext in-memory only, never echoed or written to scrollback.

## Other observations (outside the six focus areas)

The invite-consumption ordering (#2, promoted to a definite bug above) and the
multi-session registry quirks: with concurrent sessions of one account, a PHONE
ring reaches only *one* session (shared `notify` channel, `model.go:104`), and
`KICK` only closes the most-recently-registered session (`SetKick` overwrites
`entry.kick`, `registry.go:184`). Both are low-severity multi-session edge cases.

Nothing here is an emergency except the trajectory of #1 over long uptime.
Recommended fix order: **#1 (leak)** → **#2 (invite burn)** → **#3 (guard
TOCTOU)** → **#4 (DSN guard/test)**, with the rest as cleanup.
